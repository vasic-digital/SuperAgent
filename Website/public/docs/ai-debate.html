AI Debate Configuration

Configure and manage multi-agent AI debates with multiple participants, intelligent LLM fallback chains, and Cognee AI integration for enhanced reasoning and consensus building.

Overview

The AI Debate Configuration System provides a comprehensive framework for configuring and managing AI-powered debates with multiple participants, each backed by multiple LLMs with intelligent fallback chains. The system integrates with Cognee AI for enhanced response quality and consensus analysis.

Multiple Participants

Configure 2-10 AI participants with distinct roles and personalities for diverse perspectives.

LLM Fallback Chains

Each participant can have multiple LLMs with automatic fallback for reliability.

Cognee AI Integration

Enhanced responses, consensus analysis, and insights generation through Cognee AI.

Configurable Strategies

Multiple debate and voting strategies available for different use cases.

Configuration Structure

Global Configuration

Configure global debate settings including timeouts, consensus thresholds, and strategy options:

# Global AI debate settings
enabled: true
maximal_repeat_rounds: 3
debate_timeout: 300000  # 5 minutes in milliseconds
consensus_threshold: 0.75
enable_cognee: true

# Debate strategy configuration
debate_strategy: "structured"  # round_robin, free_form, structured, adversarial, collaborative
voting_strategy: "confidence_weighted"  # majority, weighted, consensus, confidence_weighted
response_format: "detailed"  # brief, detailed, structured, bullet_points

# Memory and context management
enable_memory: true
memory_retention: 2592000000  # 30 days in milliseconds
max_context_length: 32000

# Quality and performance settings
quality_threshold: 0.7
max_response_time: 30000  # 30 seconds
enable_streaming: false

Cognee AI Configuration

Configure Cognee AI integration for enhanced debate capabilities:

cognee_config:
  enabled: true
  enhance_responses: true
  analyze_consensus: true
  generate_insights: true
  dataset_name: "ai_debate_enhancement"
  max_enhancement_time: 10000  # 10 seconds
  enhancement_strategy: "hybrid"  # semantic_enhancement, contextual_analysis, knowledge_integration
  memory_integration: true
  contextual_analysis: true

Participant Configuration

Configure individual debate participants with their roles, styles, and LLM configurations:

participants:
  - name: "Analyst"
    role: "Primary Analyst"
    description: "Main analytical participant with comprehensive reasoning"
    enabled: true
    maximal_repeat_rounds: 3
    response_timeout: 30000
    weight: 1.5
    priority: 1

    # Debate behavior
    debate_style: "analytical"  # analytical, creative, balanced, aggressive, diplomatic
    argumentation_style: "logical"  # logical, emotional, evidence_based, hypothetical
    persuasion_level: 0.8
    openness_to_change: 0.3

    # Quality settings
    quality_threshold: 0.8
    min_response_length: 100
    max_response_length: 2000

    llms:
      - name: "Claude Primary"
        provider: "claude"
        model: "claude-3-5-sonnet-20241022"
        enabled: true
        api_key: "${CLAUDE_API_KEY}"
        temperature: 0.1
        max_tokens: 2000
        weight: 1.0

      - name: "DeepSeek Fallback"
        provider: "deepseek"
        model: "deepseek-coder"
        enabled: true
        api_key: "${DEEPSEEK_API_KEY}"
        temperature: 0.1
        max_tokens: 2000
        weight: 0.9

Fallback Mechanism

The system implements intelligent fallback mechanisms to ensure reliability:

  • Primary LLM Failure: Automatically tries next LLM in chain
  • Retry Logic: Configurable retry attempts per LLM
  • Timeout Handling: Respects timeout settings at multiple levels
  • Quality Fallback: Falls back if response quality is too low
  • Consensus Fallback: Continues debate if consensus isn't reached

Best Practice: Always configure at least 2 LLMs per participant to ensure fallback availability.

Programmatic Usage

Use the AI Debate system programmatically in your Go applications:

package main

import (
    "context"
    "log"
    "time"

    "github.com/helixagent/helixagent/internal/config"
    "github.com/helixagent/helixagent/internal/services"
)

func main() {
    // Load configuration
    loader := config.NewAIDebateConfigLoader("configs/ai-debate-example.yaml")
    cfg, err := loader.Load()
    if err != nil {
        log.Fatalf("Failed to load configuration: %v", err)
    }

    // Create debate service
    debateService, err := services.NewAIDebateService(cfg, nil, nil)
    if err != nil {
        log.Fatalf("Failed to create debate service: %v", err)
    }

    // Conduct a debate
    ctx, cancel := context.WithTimeout(context.Background(), 10*time.Minute)
    defer cancel()

    result, err := debateService.ConductDebate(ctx,
        "Should AI be regulated?",
        "Initial context about AI regulation")
    if err != nil {
        log.Fatalf("Debate failed: %v", err)
    }

    // Process results
    log.Printf("Debate completed in %v", result.Duration)
    log.Printf("Consensus reached: %v", result.Consensus != nil && result.Consensus.Reached)
    log.Printf("Best response: %s", result.BestResponse.Content)
}

Configuration Validation

The system performs comprehensive validation on all configuration parameters:

Global Validation Rules

Parameter Valid Range
maximal_repeat_rounds 1-10
consensus_threshold 0.0-1.0
quality_threshold 0.0-1.0
participants 2-10 participants

LLM Validation Rules

Parameter Valid Range
temperature 0.0-2.0
top_p 0.0-1.0
frequency_penalty -2.0 to 2.0
presence_penalty -2.0 to 2.0

Important: Store API keys in environment variables, never in configuration files directly.

Environment Variables

Required environment variables for AI debate configuration:

# LLM Provider API Keys
CLAUDE_API_KEY=your_claude_api_key
DEEPSEEK_API_KEY=your_deepseek_api_key
GEMINI_API_KEY=your_gemini_api_key
QWEN_API_KEY=your_qwen_api_key
OPENROUTER_API_KEY=your_openrouter_api_key
ZAI_API_KEY=your_zai_api_key

# Cognee AI Configuration (optional)
COGNEE_API_KEY=your_cognee_api_key
COGNEE_ENABLED=true

Best Practices

  • Use Environment Variables: Store API keys and sensitive data in environment variables
  • Configure Timeouts Appropriately: Set realistic timeouts based on your use case
  • Implement Fallback Chains: Always configure multiple LLMs for reliability
  • Monitor Performance: Use built-in metrics to monitor system performance
  • Test Configurations: Thoroughly test configurations before production deployment
  • Regular Updates: Keep LLM models and configurations updated
  • Security First: Follow security best practices for API key management
