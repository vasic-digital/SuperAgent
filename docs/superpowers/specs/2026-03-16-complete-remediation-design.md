## 1. Problem Statement

Comprehensive audit of the HelixAgent project identified the following categories of work required to achieve 100% completion across all dimensions:

- 4 goroutine leak risks in HTTP handlers (discovery, model_metadata, background_task, acp)

- 240+ files with `go func()` requiring shutdown verification (audit scope: all files in `internal/handlers/`, `internal/services/`, `internal/cache/`, `internal/database/`; remaining files in providers and tests are either already safe via context patterns or are test-only code)
- SpecAdapter nil receiver risk

### 1.2 Test Coverage Gaps
- `internal/adapters/background/` — 4 files, 1,024 lines, ZERO tests (critical)
- 8 packages with limited test coverage: utils (25%), knowledge (33%), planning (33%), models (67%), observability (60%), llmops (60%), selfimprove (60%), optimization (75%)
- No dedicated monitoring/metrics validation tests
- No extreme stress scenarios (10x load)

### 1.3 Security
- 7 security scanners configured but findings not analyzed/resolved
- Need comprehensive scan + remediation cycle

### 1.4 Performance
- Lazy loading opportunities not fully exploited
- Some synchronous operations in hot paths
- Unbounded concurrent operations in some subsystems

### 1.5 Dead Code (CONST-006)
- Comprehensive dead code scan required: unused exported functions, disconnected features, unreferenced adapters
- Production mock/stub/placeholder verification (CLAUDE.md rule 9)

### 1.6 Documentation & Content
- All existing docs need updates to reflect remediation changes
- Video courses need new modules for safety patterns
- Website docs pages need content refresh
- **CONSTITUTION.md (v1.0.0) and CONSTITUTION.json (v1.2.0) are stale** — must be synchronized with CLAUDE.md (v1.2.0, 26 rules) per CONST-020

### 1.7 HTTP/3 (QUIC) & Brotli Compliance
- Constitution mandates HTTP/3 as primary transport with Brotli compression
- Must verify all HTTP clients/servers comply or add compliance where missing

## 2. Design: 12-Phase Dependency-Ordered Sweep

### Rationale
Bottom-up dependency order ensures each phase builds on stable foundations. Phase 0 captures benchmark baselines and synchronizes governance documents. Safety fixes come first because tests verifying them need stable code. Security scanning needs stable code to avoid noise. Documentation comes last because it must reflect completed work.

### Feedback Loop Protocol
If a later phase (e.g., security scanning) reveals issues that belong to an earlier phase (e.g., safety), the issue is fixed in-place and the earlier phase's acceptance criteria are re-verified before continuing.

### Phase Dependency Graph
```
Phase 0 (Prerequisites) → Phase 1 (Safety) → Phase 1b (Dead Code) → Phase 2 (Tests)
    ↓
Phase 3 (Security) → Phase 4 (Performance) → Phase 4b (HTTP/3 Verification)
    ↓
Phase 5 (Monitoring Tests) → Phase 6 (Stress Tests) → Phase 7 (Docs) → Phase 8 (Content)
    ↓
Phase 9 (Validation + Container Rebuild) → Phase 10 (Commit/Push)
```

### Out-of-Scope Rules (with justification)
- **CONST-001 (Comprehensive Decoupling)**: Module extraction is already at 35 modules. No new extraction identified in this remediation.
- **CONST-016 (Unified Configuration)**: CLI agent configs are generated by HelixAgent binary. No changes needed for this remediation.
- **CONST-021 (Container-Based Builds)**: Release builds already use containers. Not affected by this remediation.

## 3. Phase 1: Safety & Concurrency Fixes

### 3.1 Critical Goroutine Leak Fixes

**File: `internal/handlers/discovery_handler.go:175-179`**
- Problem: `go func()` spawned without context cancellation
- Fix: Accept request context, create child context with timeout, pass to `Start()`
- Add `sync.WaitGroup` tracking for the goroutine

**File: `internal/handlers/model_metadata.go:205-211`**
- Problem: Uses `context.Background()` ignoring caller context
- Fix: Derive context from request context with timeout (keep 30min timeout as max)
- Add deduplication to prevent multiple concurrent refreshes

**File: `internal/handlers/background_task_handler.go:410`**
- Problem: SSE channel race window between disconnect and unregister
- Fix: Add mutex-protected `closed` flag, check before sending to channel
- Ensure channel drain on unregister

**File: `internal/handlers/acp_handler.go:121,128`**
- Problem: `stopCleanup` channel created but no `Shutdown()` method
- Fix: Add `Shutdown()` method that closes `stopCleanup` and waits for cleanup goroutine via WaitGroup

### 3.2 High-Priority Fixes

**File: `internal/cache/invalidation.go:221-241`**
- Problem: Event-driven goroutine without guaranteed cleanup
- Fix: Add `sync.WaitGroup` to track goroutine, ensure `Stop()` waits for completion

**File: `internal/database/debate_log_repository.go:419-445`**
- Problem: No tracking of cleanup worker goroutine lifecycle
- Fix: Add `running` atomic flag and WaitGroup, expose `StopCleanupWorker()` method

### 3.3 Medium-Priority Fixes

**Context propagation**: Audit top 20 most critical `context.Background()` usages in handlers and services. Replace with parent context where a request/application context is available.

**Cache expiration WaitGroup**: Add `sync.WaitGroup` to `ExpirationManager.Start()` to track `cleanupLoop` and `validationLoop` goroutines.

**SpecAdapter nil safety**: Add nil receiver checks to all methods, or return a no-op adapter instead of nil.

### 3.4 Acceptance Criteria
- All identified goroutine leaks have proper lifecycle management
- `go test -race ./internal/handlers/... ./internal/cache/... ./internal/database/...` passes
- No goroutine count increase after handler request completion (verified in tests)

## 13. Constraints & Principles

1. **Resource limits** (CONST Resource Management): All tests run with GOMAXPROCS=2, nice -n 19, ionice -c 3 (30-40% host resources). This applies to ALL phases including stress tests.
2. **No interactive processes** (Non-Interactive Execution): No sudo, no password prompts, no interactive shells. All secrets via env vars or .env files.
3. **No breaking changes** (CONST-013 Rock-Solid Changes): Every fix must preserve existing functionality
4. **SSH only for git** (CONST-018a): All git operations use SSH URLs
5. **Container orchestration via HelixAgent only** (CONST-015, CONST-015a, CONST-021): No manual docker/podman commands. All container lifecycle through HelixAgent boot process.
6. **Real data only** (CLAUDE.md rule 9): No mocks in production code. Verified in Phase 0.
7. **Constitution compliance**: All 26 mandatory rules followed. Rules verified by CONST-ID reference.
8. **Manual CI/CD only** (CONST-019): No GitHub Actions. All validation manually triggered.
9. **Mandatory container rebuild** (CLAUDE.md rule 11a): All containers rebuilt after code changes affecting them.
10. **SOLID/KISS/DRY/YAGNI** (CONST-011, CONST-012): All code changes follow software principles and appropriate design patterns.

## 14. Risk Mitigation

| Risk | Mitigation |
|------|-----------|
| Safety fixes introduce regressions | Run full test suite after each fix |
| Security scanner containers unavailable | Fall back to non-containerized scanners (gosec, go vet) |
| Stress tests crash host | Strict resource limits enforced (GOMAXPROCS=2, nice -n 19, ionice -c 3) |
| Documentation drift | Update docs immediately after each phase |
| Git conflicts | Work on single branch, commit frequently |
| Security scan reveals safety issues | Feedback loop: fix in-place, re-verify earlier phase acceptance criteria |
| Scanner requires interactive auth | All credentials via env vars; skip scanner if token unavailable |
| Phase 0 baseline capture fails | Proceed without baseline; document reason |
